    Introduction

    The Cisco PyUSB is the Python library used to program the Cisco PoE to USB adapter. When combined with the PoE to USB adapter, the Cisco PyUSB allows you to use the same USB access programs written with PyUSB to access your USB devices over a Network rather than being confined to a local USB connection.

    The Cisco PyUSB library operates consistently with the standard PyUSB Python Library and as such most of this document talks about the standard calls. As such, because the Cisco PyUSB is based upon PyUSB, this document will describe what you can do with PyUSB and will follow this with an explanation of the Cisco unique extensions at the bottom of the guide. But basically, the Cisco PyUSB provides the same features as PyUSB with the addition of the Cisco Network Interfaces. These features include:

    • 100% written in Python - this allows Python programmers to better understand how PyUSB works.

    • Platform neutrality - PyUSB version implements a frontend-backend architecture to isolate the API from system specific implementation details. The glue between the two layers is the IBackend interface. While the open source PyUSB comes with builtin backends for libusb 0.1, libusb 1.0 and OpenUSB, the Cisco Version comes with a backend that redirects to the Network interface used by the EIoT PoE to USB adapter.

    • Portability - Cisco PyUSB will run on any platform with Python >= 2.4 and ctypes along with the EIoT PoE to USB adapter.

    • Ease of Programming - because of the ease of programming with PyUSB, you can use the Cisco PyUSB to easily communicate with a USB device over the network. While USB is a complicated protocol, PyUSB (and thus Cisco PyUSB) has good defaults for most common configurations.

    • Cisco Extensions - using the backend capabilities of PyUSB, Cisco has extended the functions to allow you to find, configure, and communicate with USB devices that have been connected to your network using the EIoT DevKit via a Cisco switch.

    Logging and Debugging with PyUSB

    Every function in PyUSB will raise an exception when an error is encountered. In addition to the Python standard exceptions, PyUSB defines usb.core.USBError for PyUSB specific errors.

    You can trap and log errors in PyUSB by using the standard Python logging module. To enable PyUSB logging, you define the environment variable PYUSB_DEBUG with one of the following logging level values:

    • critical - only log critical errors
    • error - only log critical errors and errors
    • warning - log warnings, errors, and critical errors
    • info - log info, warnings, errors, and critical errors
    • debug - log everything including debug messages

    By default, logged messages are sent to sys.stderr. You can redirect log messages to a file by defining the PYUSB_LOG_FILENAME environment variable to a string that is a valid file path and all logged messages will be written to the file rather than being sent to sys.stderr.

    Although PyUSB makes USB programming less painful, it is still assumed in this guide that you have a minimal knowledge of the USB protocol. If you dont know anything about USB, and would like to know more, the original author of PyUSB recommends that you refer to the, USB Complete by Jan Axelson.

    Since the PyUSB library is low level, it is typically necessary to use a USB bus analyzer to determine the USB configuration and read/write commands.

    The PyUSB Modules

    import usb.core
dev = usb.core.find()

    Below is an overview on the PyUSB modules. Cisco PyUSB modules are under the usb package and contains the modules below. The last, USBIP is a unique backend provided by Cisco.

    Module Name Description
    core The main USB module.
    util Utility functions.
    control Standard control requests.
    legacy Module to provide compatibility with previous versions of PyUSB.
    backend Module that contains all of the built-in backends.
    USBIP Cisco backend that connects PyUSB to USB devices connected over an IP Network using the Cisco to PoE Adapter connected to a Cisco Switch

    You will import modules into your Python code using the import command. For example, to import the core module, you would use the code on the right.

    PyUSB Hello World Program

    PyUSB Hello World

    import usb.core
import usb.util

# find our device
dev = usb.core.find(idVendor=0xfffe, idProduct=0x0001)

# was it found?
if dev is None:
    raise ValueError('Device not found')

# set the active configuration. With no arguments, the first
# configuration will be the active one
dev.set_configuration()

# get an endpoint instance
cfg = dev.get_active_configuration()
intf = cfg[(0,0)]

ep = usb.util.find_descriptor(
    intf,
    # match the first OUT endpoint
    custom_match = \
    lambda e: \
        usb.util.endpoint_direction(e.bEndpointAddress) == \
        usb.util.ENDPOINT_OUT)

assert ep is not None

# write the data
ep.write('hello world')

    To the right is a simplistic program that sends the hello world string to the first OUT endpoint found.

    The first two lines of code imports PyUSB package modules. usb.core is the main module, and usb.util contains utility functions.

    The next line searches for our device and returns an instance object if it is found. If not, None is returned.

    Next, we set the device configuration for subsequent use. Note that no argument were supplied to indicate any special device configurations were needed; we simply used the default. You will see, many PyUSB functions have defaults that work for most common devices. In this case, the configuration set is the first one found.

    Next, we look for the endpoint that we are interested in. We search for it inside the first interface we have. After finding the endpoint, we send the data to it.

    Finally, we write string Hello Word to our device.

    Each of the functions called in this example will be explained in this document.

    PyUSB find function

    The find function in PyUSB is used to locate a USB device on the system. With Cisco PyUSB, the device that you locate will be on the network due to the PoE to USB adapter. Once you find the device you can initialize it and communicate to it over the network using the other PyUSB commands

    A Simple PyUSB find example

    Using the PyUSB find function

    import usb.core

dev = usb.core.find(idVendor=0xfffe, idProduct=0x0001)
if dev is None:
    raise ValueError('Our device is not connected')

    The find function part of the core module, is used to find and to enumerate devices connected to the system. For example, lets say that your device has a vendor ID equal to 0xfffe and product ID equals to 0x0001. To find it, you would use the code to the right.

    find a Specific Class of Device

    Find a Printer

    # use bDeviceClass=7 for printer
if usb.core.find(bDeviceClass=7) is None:
    raise ValueError('No printer found')

    The find function will return a usb.core.Device object representing your device. If the device is not found, it returns None. You can use any field of the Device Descriptor you need. For example, to discover if there is a USB printer connected to the network, you can use the code to the right. In the example, the bDeviceClass of 7 is the class code specified in the USB specification for a printer.

    find All Devices of a Specific Class

    Find All Printers

    # Look for all bDeviceClass=7 which are all printers
printers = usb.core.find(find_all=True, bDeviceClass=7)

# now write out the number of printers
import sys
sys.stdout.write('There are ' + len(printers) + ' in the system\n.')

    Using the code in the example to the right, you can find all printers on the system.

    find also has a parameter called find_all that defaults to false. When it is set to false or defaults to false, find will return the first device found that matches the specified criteria. However, if you give it a true value, find will instead return a list with all devices matching the criteria.

    The find_all=True Parameter

    Complete Program To Find All Printers

    import usb.core
import usb.util
import sys

class find_class(object):
    def __init__(self, class_):
         self._class = class_
    def __call__(self, device):
        # first, let's check the device
        if device.bDeviceClass == self._class:
            return True
        # ok, transverse all devices to find an
        # interface that matches our class
        for cfg in device:
            # find_descriptor: what's it?
            intf = usb.util.find_descriptor(
                                        cfg,
                                        bInterfaceClass=self._class
                                )
            if intf is not None:
                return True

        return False

printers = usb.core.find(find_all=1, custom_match=find_all(7))

    Many USB devices actually put their class information in the Interface Descriptor instead of the Device Descriptor. To accurately find all printers connected to the system, you need to transverse all configurations and all interfaces looking for a bInterfaceClass field that is equal to 7.

    The code to the right shows an elegant way to do this in PyUSB by finding all printers regardless of how they describe themselves by using the parameter custom_match.

    customer_match Combined With Other Parameters

    Combine custom_match With Other Parameters

    # find all printers that belongs to vendor 0xfffe
printers = usb.core.find(find_all=1, \
    custom_match=find_class(7), \
    idVendor=0xfffe)

    The custom_match parameter will work with any callable object that accepts the device object. When used, custom_match will return true for a matching device, and false for a non-matching device. You can combine custom_match with device fields to create more more searches. The example to the right show how this can be used to find all printers from the vender 0xfffe.

    Reading Device Attributes

    After you have found your USB device, you will next want to query it to understand its configurations, interfaces, endpoints, transfer types, and other attributes.

    The find will return a device object that you can directly read its descriptor fields as object properties. For example, if your find returned the device object dev you can read the attributes of:

    • dev.bLength
    • dev.bNumConfigurations
    • dev.bDeviceClass

    Read All of the Device Configuration

    Iterate Over the Device Configuration

    for cfg in dev:
    sys.stdout.write(str(cfg.bConfigurationValue) + '\n')

    for intf in cfg:
        sys.stdout.write('\t' + \
                         str(intf.bInterfaceNumber) + \
                         ',' + \
                         str(intf.bAlternateSetting) + \
                         '\n')
        for ep in intf:
            sys.stdout.write('\t\t' + \
                             str(ep.bEndpointAddress) + \
                             '\n')

    The code to the right shows how you would iterate over all of the fields in the device descriptor (cfg) that are in the device object (dev). The code writes each value found in the iteration to stdout. The code continues by iterating over a configuration (cfg) to access all of the interfaces (intf) and subsequently iterates over the interfaces (intf) to access their endpoints (ep) again writing all of the information discovered to stdout.

    Random Access of Device Descriptors

    Random Access of Device Descriptors

    # access the second configuration
cfg = dev[1]
# access the first interface
intf = cfg[(0,0)]
# third endpoint
ep = intf[2]

    You can also use the subscript operator to access the descriptors randomly such as in the example to the right.

    As expected in Python, the index is zero-based. But you should note, that the subscript operator in the Configuration accepts a sequence of two items (i.e., it is two dimensional). The first dimension is the index of the interface; the second dimension will access the alternate setting. Thus, to access the first interfaces second alternate setting, you would use cfg[(0,1)].

    The find_descriptor Function

    The find_descriptor Function

    import usb.util
alt = usb.util.find_descriptor(cfg, find_all=True, bInterfaceNumber=1)

    PyUSB has a powerful way to find descriptors called the find_descriptor function. The find_descriptorfunction is similar to the find function but with two exceptions:

    • find_descriptor receives as its first parameter the parent descriptor that will be searched
    • find_descriptor has no backend parameter

    In the example to the right, the configuration descriptor cfg is searched using the find_descriptor to find all alternate settings of the interface 1. The find_descriptor is in the usb.util module and will work with the custom_match parameter.

    Dealing with multiple identical devices

    When two or more identical USB devices are connected to your system, to help you differentiate them, PyUSB device objects come with two additional attributes which are not part of the USB Spec; the bus and address attributes. These attributes are populated from the PyUSB backend and if the backend does not support them, they are set to None. The attributes represent the bus number and bus address of each device respectively and can be used to differentiate two devices with the same idVendor and idProduct attributes.

    USB Device Configuration

    Set Device Configuration

    dev.set_configuration(5)
dev.set_configuration() # we assume the configuration 5 is the first one
cfg = util.find_descriptor(dev, bConfiguration=5)
cfg.set()
dev.set_configuration(cfg)

    USB devices must have their configuration set before they can be used and after they are connected. Thus, after getting your device object and before you can communicate with the device, you must issue a set_configuration request. The parameter for this request is the bConfigurationValue for the configuration you are interested in. Most USB devices have no more than one configuration, and tracking the configuration value is a cryptic part of the specification. With PyUSB, you can just issue a set_configuration call with no arguments and it will figure it out for you. What PyUSB does is to set the first configuration found (and thus, if your device has just one, you dont need to worry about the configuration value at all).

    The code to the right shows how you can set a device configuration through standard PyUSB set_configuration functions. In the example, the code to the right sets the bConfigurationValue field equals to 5. In this example the Configuration object is used as a parameter to set_configuration in which it has a set method to configure itself as the current configuration.

    USB Interface Alternative Settings

    set_interface_altsetting Function

    try:    
    dev.set_interface_altsetting(interface = 0, alternate_setting = 0)
except USBError:
            pass

    The USB devices also have a concept of alternate settings for an Interface. Each device can have only one active configuration at a time yet each configuration may have more than one interface. Further, you can use all of the interfaces at the same time and thus it is best to think of an interface as a logical device.

    For example, a multi-function printer is simultaneously a printer and a scanner. Yet our multi-function printer is a single USB configuration with two interfaces; one interface for the printer and one interface for the scanner. A device with more than one interface is called a composite device in the USB Specification. When you connect your multifunction printer via USB, the Operating System loads two different drivers: one for each logical peripheral or interface in the USB configuration.

    An interface can have one or more alternate settings. Although you can have multiple, alternative settings for an interface only one alternate setting is active at a time. For example, USB spec says that a device cannot have an isochronous endpoint in its primary alternate setting, so a streaming device must have at least two alternate settings, with the second one having the isochronous endpoints. An interface with just one alternate setting does not need to be explicitly set since it is the only one. You can select an interface alternate setting through the set_interface_altsetting function.

    The USB spec says that a device is allowed to return an error if it receives a SET_INTERFACE request for an interface that has no additional alternate settings. Thus, if you are not sure if the interface has more than one alternate setting or if it will accept a SET_INTERFACE request, it is best to call set_interface_altsetting inside a Python try-except block as shown in the example.

    Using an Interface Object as a Parameter

    intf = find_descriptor(...)
dev.set_interface_altsetting(intf)
intf.set_altsetting() # wow! Interface also has a method for it

    An Interface object can also be used as parameter to the set_interface_altsetting function and when you do so, the interface and alternate_setting parameters will be automatically inferred from bInterfaceNumber and bAlternateSetting fields. The code to the right shows an example of this.

    The Interface object must belong to the active configuration descriptor when used as a parameter.

    Communicating With USB Devices

    After you have found and configured your USB device, you will want to communicate with it.

    USB communication is done in one of four transfer modes:

    • bulk
    • interrupt
    • isochronous and
    • control transfer

    This document will assume that you know the basics of the USB transfers modes and will not explain the purpose and the differences between them.

    Control transfer is the only transfer mode that sends and receives structured data, the others just send and receive raw data to/from the USB device. Because of this difference, control transfer mode has its own function to deal with control transfer communications while the other three modes share a common function.

    Control Transfer Communications

    msg = 'test'
assert dev.ctrl_transfer(0x40, CTRL_LOOPBACK_WRITE, 0, 0, msg) == len(msg)
ret = dev.ctrl_transfer(0xC0, CTRL_LOOPBACK_READ, 0, 0, len(msg))
sret = ''.join([chr(x) for x in ret])
assert sret == msg

    Control transfer communication is done via the ctrl_transfer method. It is used for data transfer in both directions with the device; the direction of the transfer will be determined from the bmRequestType parameter.

    The example to the right show how you would issue a control_transfer.

    In the example, it is assumed that your device implements two custom control requests that act as a loopback pipe. What is written with the CTRL_LOOPBACK_WRITE message can also be read with the CTRL_LOOPBACK_READ message.

    The first four parameters in the ctrl_transfer parameters are:

    • bmRequestType,
    • bmRequest,
    • wValue, and
    • wIndex

    These parameters are fields of the standard control transfer structure in the USB specification. The fifth parameter is either the data payload for an OUT transfer or the number of bytes to read in an IN transfer.

    The data payload can be any sequence type that can be used as a parameter for the array __init__ method in Python. If there is no data payload, the parameter should be None (or 0 in case of an IN transfer). There is a six, optional timeout parameter for the transfer. If you dont supply a timeout value, then the default timeout for the device will be used. In an OUT transfer, the return value is the number of bytes really sent to the device. In an IN transfer, the return value is an array_ object with the data read.

    Bulk, Interrupt, and Isochronous Communications

    msg = 'test'
assert len(dev.write(1, msg, 100)) == len(msg)
ret = dev.read(0x81, len(msg), 100)
sret = ''.join([chr(x) for x in ret])
assert sret == msg

    For bulk, interrupt, and isochronous transfer modes, you simply use the write and read methods. There is no need to worry about the transfer type, it is automatically determined from the endpoint address. The code to the right is a loopback example assuming the we have a loopback pipe in the endpoint.

    In the write and read functions, the first and third parameters are the endpoint address and timeout, respectively. The second parameter is the data payload for the write function and the number of bytes to read for the read function. The returned data is either an instance of the array_ object for the read method or the number of bytes written for the write method.

    Instead of the number of bytes to be read, the read and ctrl_transfer read method can be passed an array_ object into which the data will stored upon read. When used this way, the number of bytes to read will be the length of the array times the array.itemsize value.

    As in ctrl_transfer, the timeout parameter is optional. When the timeout is omitted, the Device.default_timeout property is used.

    Additional Topics

    Additional Modules

    In addition to the transfers functions, the module usb.control also includes functions that implement the standard USB control requests.

    The usb.util module has the get_string function that can be used to return string descriptors instead of values.

    Backend Parameter for find Function

    The find function has one more parameter that has not yet covered. It is the backend parameter. If you dont supply the backend parameter, the system will automatically used one of the built-in backends.

    Each backend is an object inherited from usb.backend.IBackend and the backend is responsible for implementing the operating system specific USB stuff. PyUSB has the built-ins of libusb 0.1, libusb 1.0 and OpenUSB. For the Cisco PyUSB, a backend has been written specifically to support the USB access of devices over an IP network that have been connected with the Cisco PoE to USB adapter.

    Python Memory Management

    Python provides automatic memory management. This means that the Python virtual machine automatically decides when to release objects from the memory. Under the hood, PyUSB automatically manages all the low level resources it needs (e.g., interface claiming, device handles, etc.) and thus most users do not need to worry about memory management.

    However, because of the nondeterministic nature of automatic object destruction of Python objects, a developer cannot predict when unused, allocated resources will be released and thus, you might want to allocate and free the resources deterministically. The usb.util module provides you with a set of functions if you want to take control of your own resource management.

    To claim and release interface objects manually, you can use the claim_interface and release_interface functions respectively.

    • The claim_interface will claim the specified interface if the device has not yet done it. If the device has already claimed the interface, then a call to the claim_interface function will do nothing.

    • The release_interface will explicitly release the specified interface if it is claimed. If the interface is not claimed, it does nothing.

    To free all resources allocated by the device object (including interfaces claimed), you can use the dispose_resources function which will release all resources allocated to a device and restores the device object (but not the device hardware itself) to the state it was at the time when the device was discovered by the find function.

    Specifying Python Libraries by Hand

    import usb.core
import usb.backend.libusb1

backend = usb.backend.libusb1.get_backend(find_library=lambda x: /
    "/usr/lib/libusb-1.0.so")
dev = usb.core.find(..., backend=backend)

    A backend, such as Ciscos USBIP, is a wrapper in a shared library which implements the USB access API. By default, the backends use the find_library() ctypes function. On Linux and other Unix-like Operating Systems, the find_library function will run external programs (e.g., /sbin/ldconfig, gcc and objdump) to find the library file.

    On some systems that have the external programs missing or disabled, the required external functions may not be available for use and thus the system is not able to find the Python Libraries. To overcome this limitation, PyUSB allows you to supply a custom find_library() function.

    The code to the right shows how this is done. Notice that the find_library argument for the get_backend() function, supplies a function that is responsible to find the correct library for the backend; in this case it provides an explicit path to the module.

    Cisco Extensions to PyUSB

    Using the backend module capabilities of PyUSB, Cisco created the USBIP backend which allows a program written using PyUSB to find, configure, and communicate with a USB device connected via a Cisco switch running the EIoT USBIP server.

    For the PyUSB library to connect to a remote USB device using PoE to USB DevKit, your Python program needs to know the hostname or IP address of the Cisco switch that the device is connected through. You specify the host Switch using the Python environment variable USBIP_SERVER. For example, the following line of code will point the PyUSB library to the Cisco switch at IP address 10.1.2.3:

    os.environ["USBIP_SERVER"] = 10.1.2.3

    Cisco PyUSB also allows the find function to find a USB device by using the MAC address of the PoE to USB DevKit adapter. This can be useful for attaching to a specific USB device when the normal PyUSB find() mechanism is ambiguous. For example, the following code will attach to the USB device connected to the PoE to USB adapter with MAC address 00-22-bd-cf-36-00:

    dev = usb.core.find(address=int("0022bdcf3600", 16)

    Cisco PyUSB Hello World Program

    Cisco PyUSB Hello World

    import usb.core
import usb.util

# find device with Mac address of the PoE to USB adapter
dev = usb.core.find(address=int("0022bdcf3600", 16))

# old code that used idVendor and idProduct to locate the device
# dev = usb.core.find(idVendor=0xfffe, idProduct=0x0001)

# was it found?
if dev is None:
    raise ValueError('Device not found')

# set the active configuration. With no arguments, the first
# configuration will be the active one
dev.set_configuration()

# get an endpoint instance
cfg = dev.get_active_configuration()
intf = cfg[(0,0)]

ep = usb.util.find_descriptor(
    intf,
    # match the first OUT endpoint
    custom_match = \
    lambda e: \
        usb.util.endpoint_direction(e.bEndpointAddress) == \
        usb.util.ENDPOINT_OUT)

assert ep is not None

# write the data
ep.write('hello world')

    At the beginning of the document, we showed you a standard PyUSB Hello World program. To the right is the same program modified to work with a device that is connected via the PoE to USB adapter. Notice that the program now uses a different find() function to find the device using the MAC address of the adapter. The program then continues unaltered in configuring the device, opening it with find_descriptor and writing to the device using the write() function.
